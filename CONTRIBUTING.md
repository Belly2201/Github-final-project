# Guía de Contribución

> Versión 1.0 — Última actualización: 10-11-2025

## Propósito

Agradecemos todas las contribuciones a este proyecto. Ya sea que se trate de reportes de errores, correcciones, mejoras en la documentación, nuevas ideas o características, tu aporte ayuda a mantener y mejorar la calidad del código y la comunidad.

## Tipos de contribuciones aceptadas

* **Reportes de errores (bugs):** Describe el comportamiento esperado, el comportamiento actual y los pasos para reproducir el problema. Si es posible, incluye capturas de pantalla o enlaces relevantes.
* **Correcciones (bug fixes):** Envía un *pull request* (PR) con una descripción clara de la corrección y cómo fue probada.
* **Mejoras de documentación:** Corrige errores tipográficos, amplía explicaciones o actualiza ejemplos obsoletos. Incluso las pequeñas mejoras son valiosas.
* **Mejoras o nuevas funciones (enhancements):** Antes de implementar una gran mejora, abre un *issue* para discutir la propuesta con los mantenedores.
* **Ideas o sugerencias:** Si tienes una propuesta general o idea para el proyecto, compártela abriendo un nuevo *issue* con la etiqueta `idea` o `enhancement`.

## Flujo de trabajo de contribución

1. **Haz un fork** del repositorio.
2. Crea una rama para tu cambio:

   ```bash
   git checkout -b feature/nueva-funcion
   ```
3. Realiza tus modificaciones con buenas prácticas de código y mensajes de commit claros.
4. Asegúrate de que las pruebas y validaciones pasen correctamente.
5. Abre un *pull request* hacia la rama principal (`main` o `develop` según aplique) con una descripción detallada.

## Revisión y aprobación

* Todos los PRs serán revisados por al menos un mantenedor.
* Los comentarios de revisión deben ser tratados con respeto y empatía.
* Se podrá solicitar que realices ajustes antes de la fusión.
* Las contribuciones que no sigan las guías o el estilo del proyecto podrán ser rechazadas.

## Estilo de código

* Sigue las convenciones establecidas en el archivo `README.md` o en la guía de estilo específica (`.editorconfig`, `eslint`, `prettier`, etc.).
* Mantén la consistencia del formato, nombres de variables y estructura de carpetas.
* Incluye comentarios claros cuando el código no sea obvio.

## Reportes de seguridad

Si encuentras una vulnerabilidad de seguridad, **no abras un issue público**. En su lugar, contacta directamente al equipo de mantenimiento.


Nos comprometemos a responder en un plazo razonable y a manejar los reportes con confidencialidad.

## Código de conducta

Todas las contribuciones deben cumplir con el [Código de Conducta](./CODE_OF_CONDUCT.md). Cualquier comportamiento irrespetuoso o discriminatorio será motivo de moderación o bloqueo.

## Reconocimiento

Tu participación será reconocida en el archivo `CONTRIBUTORS.md` (si aplica). Cada contribución, por pequeña que sea, es una parte importante del crecimiento del proyecto.

---

### Cómo empezar

* Lee primero el `README.md` para comprender el objetivo del proyecto.
* Verifica los *issues* abiertos antes de crear uno nuevo.
* Usa etiquetas apropiadas (`bug`, `enhancement`, `documentation`, `question`, etc.).
* Sé paciente y respetuoso durante el proceso de revisión.

> **¡Gracias por contribuir!** Tu colaboración hace que este proyecto crezca y mejore para todos.
